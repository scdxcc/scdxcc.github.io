    mybatis-plus使用不当导致的oom

    1.现象

        还是在这个创业公司，一个新上线几天的项目，高峰期突然收到异常报警，线上出现oom异常，jmap 发现内存被占满，查看gc log 发现oom之前频繁Full GC。

    2.排查

    2.1 之前配置了启动参数，发生oom时，自动dump 堆的快照。


    路径是 ：/root/logs/crm_customer_java.hprof ，gc log 也是在这个目录

    2.2 使用mat 工具分析堆dump 文件。

    2.2.1 把dump的堆快照下载到本地。

        之前试过下载测试环境的，发现文件很大，本地机器的网络比较慢。剑圣压缩了一下，发现虽然是二进制文件，压缩率还是挺高的，1.2G压缩完不到300M。以后下载的时候先压缩，然后再下载到本地分析。

    2.2.2 下载mat 。

    下载地址：https://www.eclipse.org/mat/downloads.php
    注意：mac上有bug,解压完双击打开会报错，需要到包目录里面打开。

    如果导入dump文件报这个错：
    Dominator tree not available. Open the Dominator Tree or delete indices and parse again.
    可以重新建个文件夹把快照文件重新复制过去，然后重新打开一次。

    2.2.3 打开mat，选择要分析的dump 文件 。

    右上角，File ->open file ,选择对应的文件就可以。如果堆快照大于1G，改一下mat的最大堆内存参数，mat 默认 -Xmx 是1024M。路径：

    2.2.4 利用mat 分析堆dump 文件 。

    1.总览


    会显示一个总体的概况，最大的几个对象的占比，像这个少量的一些类的对象占用大量的内存，会直接会提示检测到泄漏。

    2. histogram页面

    这里会直观的展示所有的类的对象的个数和直接，间接占用的内存大小，最上面上面支持正则表达式查询，比如查我们自己的包下的对象。

    发现 com.wlc.crm.customer.entity.OperationRecordEntity 这个类有42万个对象，占用内存383m.

    3. 查看引用链

    排除软引用，弱引用，虚引用，只看强引用

    打开后，会发现主要是被tomcat线程池中的三个线程引用，说明是在接口请求中发生的。

    4. 通过日志时间和线程名去排查日志。

    可以找到出问题的sql,因为之前加过日志的trace_id,所以根据这个trance_id 1560944743854 可以很方便的查找调用链，找到问题的源头是 /xxx/clue/statisticsForMenu 这个接口。

    5. 定位问题代码，com.baomidou.mybatis-plus框架的副作用。

    1. 发现问题代码


    看代码，调用的是com.baomidou.mybatisplus的getOne方法，第一感觉没有什么问题，但是getOne查询一条为什么会把所有的结果查出来呢。带着这个问题继续往下看。
    queryWrapper里面只有一个条件，就是接口传进来的 timeStamp=0

    查了一下数据库，这个条件下会有 差不多30 万 条数据。如果同时有多个并发的请求，就会导致oom,而且这个表的数据量增长比较快，如果不处理，后面oom的间隔会越来越快

    2.com.baomidou.mybatis-plus框架的副作用
    数据库有30万条数据，但是使用com.baomidou.mybatis-plus框架的getOne方法,还是查出了所有的数据。带着这个问题，看了一下它的源码。

    这个框架的源码也比较简单，到这里就明白了，getOne默认情况下是当查询结果有多个 result 时不抛出异常。会把所有的结果查出来，然后取第一条。这样就引发了这次的问题。

    3.结论与思考

    1. 写代码尽量要严谨，尽量少的使用一些封装过于完善的框架，可能会隐藏一些细节，比如之前很多漏洞的struts。不太熟悉的情况下使用会产生一些很严重的副作用。

    2. 一定要避免没有limit的全量查询，对于查询大量数据处理的场景，一定要分批次处理，对内存使用最好有个比较初步的预估。
