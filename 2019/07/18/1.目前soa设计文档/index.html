    当前soa设计

    一 soa目前功能。

    1. 整体架构

    2. 模块功能

    1 服务提供者在启动时，向注册中心注册自己提供的服务和服务ip。
    2 服务消费者在请求应用时，向注册中心订阅请求的服务。
    3 注册中心返回服务提供者地址列表和相关配置信息给消费者，如有变更，注册中心将基于长连接推送变更数据给消费者。
    4 服务消费者，从提供者地址列表中，基于健康过滤和软负载均衡算法，选一台提供者进行调用，如果调用失败，可重试。服务消费者，记录每次请求时间，供Monitor分析统计调用次数和调用时间等。

    3.soa sdk运行原理

    3.1 provider sdk

    1.server启动时，通过HostRegisterListener向zk注册中心注册服务相关配置数据（路径”/hotel_soa/configRoot/“ + productName + “/“ + serviceName + “/serverHosts/“ + idcName + “/active”）注册活跃节点。

    2.注册methods(（路径”/hotel_soa/configRoot/“ + productName + “/“ + serviceName + “/serverHosts/“ + idcName + “/methods”+method.getMethodLocation()).

    3.并监听zk连接状态事件。如果会话过期导致zk异常掉线后，重新注册服务相关配置。并且在连接断开和重连后，发送报警。

    3.2 client sdk

    1 客户端向zk注册中心注册，初始化服务相关配置数据，并订阅zk相关路径更改通知。
    2 请求主机健康过滤：将配置的可用主机，去除不健康的主机，或是从中选择一台可进行重试的不健康主机。
    3 运用随机，轮询或最小连接数选择算法，从健康主机列表中选取一台
    4 httpRequest请求.
    5 调用请求计数
    6 失败自动切换，当出现失败，重试其它服务器；如果是业务异常，直接返回配置的默认值
    返回结果，记录调用日志。

    二. sidecar设计

    一.需求分析

    1. 需求

    需要支持 nodejs作为客户端对原有的 soa 服务实现调用。

    2. 分析

     由于之前的soa以java sdk的形式提供给应用方，包括寻址，负载均衡，代理调用等核心逻辑都在sdk中实现。这次不在调用端（nodejs）嵌入任何sdk。所以sidecar理论上只对外提供http服务，暴露http接口。服务调用方通过http进行调用，相关参数或者元数据在header或者参数中透传。

    二. 整体架构

    时序图

    1.运行过程

    1. sidecar提供http服务。consumer请求sidecar的invoke接口，传入自身的productname,servicename以及其他的元数据，和要调用的provider端的 productname,servicename等数据以及请求参数数据。
    2. sidecar发现是第一次调用，会通过控制中心向zookeeper中注册client相关的信息。同时从控制中心拉取provider的相关信息，存在本地缓存，并且启动定时任务进行provider的缓存数据的更新。
    3. sidecar从控制中心或者本地缓存获得serivice相关的provider数据，通过自身的负载均衡策略选择一个可用的host进行远程调用。
    4. 将远程调用的结果返回给client端，一次调用结束。

    2. 存在的问题

    1. 原来的soa consumer sdk 依赖于zookeeper的临时节点实现consumer的存活检测。这次改造consumer无法直接与zk交互，无法实现consumer的存活检测。解决的方法是sidecar和consumer之间通过心跳检测，但是需要consumer端支持。由于consumer的alive不影响主流程，所以暂时不处理。
    2. 原来的 soa consumer sdk通过zk的watch机制实现配置的更新。sidecar不与zk直接交互，所以通过定时任务轮询来实现配本地的配置的更新。
